=== Definition of Ready
// Listen Sie hier die Kriterien auf, die ein Product Backlog Item (z.B. User Story) erfüllen muss, damit es bei der Sprint-Planung berücksichtigt werden kann

Ein Product Backlog Item (z. B. eine User Story) kann für die Sprint-Planung berücksichtigt werden, wenn folgende Kriterien erfüllt sind:

* Die User Story ist klar formuliert, inklusive Akzeptanzkriterien.
* Die User Story wurde vom gesamten Team verstanden.
* Der Nutzen (Business Value) ist bekannt.
* Das Item wurde geschätzt (z. B. in Story Points).
* Es ist klein genug, um innerhalb eines Sprints umgesetzt zu werden.
* Alle relevanten Informationen, Anhänge oder Designs liegen vor.
* Abhängigkeiten sind bekannt und geklärt.
* Die User Story ist priorisiert.

=== Definition of Done
// Listen Sie hier alle Kritierien auf, damit die Aufgabe eines bestimmten Typs (z.B. Task, User Story) als abgeschlossen gilt.

Eine User Story oder ein Task gilt als abgeschlossen, wenn folgende Kriterien erfüllt sind:

* Alle Akzeptanzkriterien sind erfüllt.
* Der Code ist vollständig implementiert und getestet (Unit- und ggf. Integrationstests).
* Der Code wurde von mindestens einer Person im Team reviewed.
* Alle automatisierten Tests laufen erfolgreich durch.
* Die Funktionalität wurde manuell getestet (falls erforderlich).
* Es gibt keine bekannten offenen Bugs.
* Die Änderung ist dokumentiert (z. B. in der technischen Doku oder im Changelog).
* Die Implementierung ist in den Hauptbranch integriert.
* Die Software ist bereit für den Release bzw. das Deployment.
* Der Product Owner hat die Umsetzung abgenommen (falls vorgesehen).
